BFF 集成模式 + Docker 部署

Fastify BFF 对接 SSO 的标准模式 + 容器编排

BFF 集成代码骨架

每个 BFF 只需要 3 个核心模块：
1. 登录守卫（中间件）
// 未登录 → 302 重定向到 SSO /authorize
req.headers.cookie?.session_id
  ? redis.get(session) → 有效 → next()
  : 302 → sso-hub.com/authorize?client_id=xxx&redirect_uri=...
2. 回调处理（/callback 路由）
// 收到 auth_code → 服务端换 token
POST sso-hub.com/token { code, client_id, client_secret }
→ { access_token, id_token, refresh_token }
→ redis.set(session, { tokens, user })
3. 受保护页面
// session 中有 id_token → 解码展示用户信息
GET /  →  返回业务 A 首页（含登录用户头像、昵称）
GET /logout  →  清除本地 session + 302 sso-hub.com/logout

项目目录结构

sso-hub/
├── packages/sso-server/  （NestJS，认证中心）
├── packages/bff-a/       （Fastify，业务 A）
├── packages/bff-b/       （Fastify，业务 B）
├── docker-compose.yml
├── traefik/          （动态配置）
└── docs/superpowers/  （设计文档）

Docker Compose 拓扑

network: public
  traefik ← 80:80, 443:443
  sso-hub  ← labels: sso-hub.localhost
  bff-a    ← labels: app-a.localhost
  bff-b    ← labels: app-b.localhost
network: internal
  sso-hub
  bff-a
  bff-b
  postgres  ← volume: pgdata
  redis      ← volume: redisdata

关键环境变量

SSO Server:
DATABASE_URL=postgresql://...
REDIS_URL=redis://redis:6379
JWT_PRIVATE_KEY / JWT_PUBLIC_KEY
GITHUB_CLIENT_ID / GITHUB_CLIENT_SECRET
BFF A / B:
SSO_URL=http://sso-hub:3000
CLIENT_ID / CLIENT_SECRET
REDIS_URL=redis://redis:6379
